# Лабораторна робота №7

## Тема
Розгортання вебзастосунку в хмарі AWS: EC2 (Ubuntu), RDS (PostgreSQL), об’єктне сховище S3, зворотний проксі Nginx, процес-менеджер PM2.

## Мета
Опанувати базову схему «веб-сервер + БД + статика в S3»: мережеві групи безпеки, підключення RDS лише з EC2, публічний доступ до HTTP, перевірка з’єднання з БД, налаштування Nginx як reverse proxy до застосунку на Node (Vite preview) під PM2.

## Регіон і загальні припущення
- Регіон AWS: **eu-north-1** (Stockholm) — використовуйте один і той самий регіон для EC2, RDS і S3.
- Публічна IP-адреса EC2 може змінитися після зупинки/запуску інстансу, якщо немає Elastic IP — у звіті вказуйте актуальну з консолі.

## Що зроблено в хмарі (огляд)
1. **IAM**: користувач з програмним доступом; ключі доступу зберігаються локально й **не потрапляють у git**.
2. **Security Groups**:
   - для EC2: SSH (рекомендовано лише з вашої IP), HTTP/HTTPS з інтернету за потреби;
   - для RDS: PostgreSQL (порт **5432**) лише з SG EC2; публічний доступ до RDS — **вимкнено**, якщо так вимагає варіант.
3. **RDS PostgreSQL**: інстанс БД у тій самій VPC/підмережах; endpoint копіюється з консолі RDS **точно** (помилка в імені дає помилку DNS).
4. **EC2 (Ubuntu)**: встановлення Node.js, клонування репозиторію з застосунком (лаб. №1), збірка (`npm install` / `npm run build` за фактичними залежностями).
5. **PM2**: запуск production/preview сервера (наприклад Vite preview на порту **4173**).
6. **Nginx**: `proxy_pass` з порту **80** на локальний порт застосунку (4173).
7. **S3**: бакет для статики; за потреби політика на читання об’єктів і CORS; перевірка публічного URL об’єкта.

## Підключення до RDS з EC2
На EC2 встановити клієнт PostgreSQL (`postgresql-client`), далі:

```bash
psql "host=<RDS_ENDPOINT> port=5432 dbname=<DB_NAME> user=<DB_USER> sslmode=require"
```

Перевірки: `\conninfo`, `SELECT version();`  
(різниця мажорних версій клієнта та сервера може давати попередження — це очікувано.)

## Структура у репозиторії
- Застосунок залишається в `lab-1-setup/` (як у попередніх лабораторних).
- Конфігурація сервера (Nginx, PM2) на EC2 задається вручну на машині; у цьому репозиторії зберігається лише **опис лабораторної** (цей файл).

## Секрети та git
**Не комітити:** файли `.pem`, `.env`, паролі БД, access key / secret key AWS, `terraform.tfvars` з токенами.  
Кореневий `.gitignore` вже ігнорує `.env*` та чутливі Terraform-файли.

## Обов’язкові скріншоти для звіту (приклад)
- **Рис. 1**: EC2 — інстанс (тип, SG, публічна IP).
- **Рис. 2**: Security Groups — правила для EC2 та RDS.
- **Рис. 3**: RDS — статус Available, endpoint, публічний доступ вимкнено.
- **Рис. 4**: Термінал EC2 — успішне `psql` і `SELECT version();`.
- **Рис. 5**: Nginx — активний сервіс; `curl -I http://127.0.0.1` — **200 OK**.
- **Рис. 6**: PM2 — список процесів застосунку.
- **Рис. 7**: S3 — бакет, завантажений об’єкт, відкриття Object URL у браузері.

## Висновок (коротко)
Розгортання на EC2 з reverse proxy дозволяє опублікувати застосунок у власній інфраструктурі; ізоляція RDS через SG зменшує поверхню атаки; S3 зручний для статичних файлів і публічних посилань. Після перезавантаження EC2 варто перевірити PM2 (`pm2 list`, за потреби `pm2 save`) та Nginx.
